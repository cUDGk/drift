# DRIFT — 構想

## きっかけ

ChatGPT や Claude は便利だけど、会話内容は必ず運営側に渡る。
ロールプレイ、微妙に人に言いづらい相談、書きかけの創作 — クラウドに残したくない会話は現実にある。

じゃあ端末内の LLM を使えば済む話、と思って llama.cpp を回してみると、今度は別の壁にぶつかる。
手のひらサイズのモデル (3B〜7B くらい) はコンテキストが狭くて、会話が長くなるとすぐ破綻する。キャラが直前の話を忘れる。設定と矛盾する台詞を言い出す。ロールプレイが続かない。

DRIFT はこの二つ — **プライバシーの無さ** と **長期一貫性の無さ** — を同時に解決することを目標にする。

## 三つの方針

### 1. 外に出さない

- 推論は端末の中で完結させる (llama.cpp / NDK / arm64-v8a)
- 会話・キャラ設定・記憶はアプリ専用領域に保存し、外部に送らない
- モデルを取り込む時以外、通信しない
- テレメトリ・クラッシュレポート・分析 SDK は入れない

### 2. 記憶は使う人のもの

会話履歴もキャラの記憶も、要約した中身も、全部ユーザー側から見えて、編集できて、削除できる。

「いい感じに忘却してます」「重要な会話だけ残してます」みたいなブラックボックスを作らない。記憶の中身に直接手を入れられる権限を、使う人が最初から持っている状態にしておく。

キャラの挙動がおかしくなった時、自分で直せない設計は根本的に破綻する。

### 3. モデルは差し替え可

- GGUF ならなんでも読めるようにする
- 用途別のモデル (軽量 / ロールプレイ / 専門) を切り替えて使える
- 今は 3B〜7B クラスを前提にするけど、もっと軽くて賢いモデルが出てきた時に本体を大きく書き直さずに乗り換えられる形にしておく

## コアの仕掛け: 階層化メモリ

狭いコンテキストで長い会話を成立させるため、記憶を何層かに分けて扱う。

| 層 | 役割 |
|---|---|
| キャラ記憶 | 会話で判明したキャラ / 自分の情報、関係性、重要イベントを構造化して常に保持する |
| 直近ターン | 最近の発言をそのまま (高解像度で) 持つ |
| 中期要約 | 古くなったターンを時系列で圧縮して積む |
| 長期アーカイブ | さらに古い要約をインデックス化、必要になった時だけ呼び戻す |

各層にトークン予算を振って、合計がコンテキスト窓に収まるよう自動調整する。各層の具体的な容量、更新タイミング、要約のやり方は実装フェーズで詰める。

## キャラクターとペルソナ

SillyTavern 系のキャラクターカード (名前・説明・性格・シナリオ・会話例) と、自分側のペルソナ (呼ばれたい名前・一人称・口調) を別々に管理する。

キャラごとに「好みのモデル」を紐付けておけば、選んだキャラに応じて裏でモデルが切り替わる。

## いま作らないもの

スコープ外にするもの:

- クラウドモデルへのフォールバック
- 複数端末間の同期
- 音声入出力、画像生成との連携
- アカウント / ログイン
- マルチユーザー

端末で軽くて賢いモデルが普通に使えるようになった時に、核の部分 (ローカル推論 + 階層化メモリ + キャラ管理) がちゃんと動いている状態を優先する。

## 似たプロジェクトとの立ち位置

- **SillyTavern** — 機能は豊富だけど基本はクラウド LLM 前提で、オンデバイス運用は弱い
- **ChatterUI / MLC Chat** — オンデバイス推論は動くけど、長期記憶やキャラ管理が素朴
- **DRIFT** — ここに「ローカル推論 + 長期記憶 + キャラ管理 + モバイル最適化」の組み合わせで入る

## 将来起きる分岐

今は 3B〜7B の量子化モデルを前提に組むけど、以下が現実になったら設計の一部は書き直す想定:

- 1B 台で今の 7B 相当の品質が出るモデルが来る → 各層のメモリ予算を大幅に広げられる
- MoE のオンデバイス実装が実用になる → 用途別モデルの自動切替の粒度が細かくできる
- 端末向けベクトル検索が軽くなる → 長期アーカイブの想起精度を底上げできる

今回の実装は、こういう変化に乗り換えやすい形を意識して組む。
